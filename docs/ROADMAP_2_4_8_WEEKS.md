# AI Novel 真实路线图（2 周 / 4 周 / 8 周）

> 创建时间：2026-05-10  
> 依据：当前工作区源码、`docs/STATUS.md`、`design.md`，以及 2026-05-10 实测 `npm run verify` 通过。  
> 历史规划文档（PROGRESS / AUDIT / TASKS / TECHNICAL_PLAN 等）已归档至 `docs/archive/`。
> 目标：从“可演示、可内部试用的 AI 小说写作 MVP”，推进到“结构清晰、体验可靠、可持续扩展的创作工作台”。

> **进度（2026-05-10 阶段 3 完成）**：8 周路线的三个阶段（M1.x / M2.x / M3.x）均已交付。阶段 3 实施时把优先级重排为「版本恢复 → 导出中心 → retrieval 可视化 → UI 降噪 → 乐观锁」，对应 ROADMAP 节段 §五.2/3/1/5/6，详见 `docs/STATUS.md` 的"长篇可靠性 + 版本 + 导出"章节；阶段 3 的 dirty 标脏（M3.1 一部分）与独立导出页面（M3.3 §五.3 完整版）按下进入 backlog。

## 一、当前基线

截至 2026-05-10，当前代码已经具备以下现实基础：

- `Onboarding` 5 步开书链路已形成闭环：session、logline、questions、Bible SSE、Review、Finalize。
- 多章节编辑器可用：自动保存、手动保存、章节切换保护、AI 起草、Critic、State Diff、导出、版本列表均已接入。
- 安全基础已具备：ownership、admin-only 模型配置、`/api/healthz/llm` 保护、API key 加密、基础限流、审核策略化。
- 长篇记忆底座已存在：Story State、VolumeSummary / NovelSummary、MemoryChunk、pgvector 检索、后台 job queue。
- 工程主干可验证：`npm run verify` 当前通过，CI 已运行 lint/typecheck/test/build。

这意味着项目已经不是“从 0 到 1”的问题，而是“从可用 MVP 到真正产品”的问题。

当前最明显的缺口不是底层能力缺失，而是：

- 产品信息架构还不完整。
- 写作工作流仍偏“技术能力拼装”，不是“创作者友好型闭环”。
- 长篇记忆和 AI 能力虽然接上了，但可靠性、可见性和交互层还不够成熟。

---

## 二、路线图原则

后续 8 周内，优先级应严格按以下顺序推进：

1. 先补产品主干，不继续堆散点能力。
2. 先把高频写作闭环做顺，再扩页面范围。
3. 先做“用户可感知的可靠性”，再做更复杂的智能性。
4. 不把“已有 API”误判成“已有完整功能”。
5. 每个阶段都必须有可验收的 UI、API、测试和文档结果。

---

## 三、2 周目标：把 MVP 收敛成可稳定内测的写作工具

### 阶段目标

把当前“能用但有拼装感”的系统，收敛成一个可以稳定内测的单人写作工具。

### 本阶段必须完成的事

#### 1. 补齐项目级信息架构最小骨架

新增最基本的项目层页面，而不是把所有能力继续塞进编辑器：

- 新增 `/novels/[id]` 小说详情页
- 新增详情页内的基础导航入口
- 明确区分：
  - 小说概览
  - 角色 / 世界观
  - 大纲 / 章节
  - 进入写作

**原因**

当前系统的最大产品问题是：用户只能“进编辑器”，但很难先理解项目结构。

#### 2. 把 Bible 侧栏能力拆成真正可管理的项目信息页

优先拆出 3 个最关键模块：

- 角色管理
- 世界观设定
- 大纲管理

第一阶段不追求复杂交互，但至少要做到：

- 有独立页面
- 可查看完整内容
- 可编辑并保存
- 与现有 `BibleDraft` 同步

#### 3. 重做 AI 起草交互，停止“直接覆盖正文”

把 AI 起草从“危险覆盖型”改成“候选稿型”：

- AI 生成结果先进入候选区
- 用户可选：
  - 覆盖正文
  - 追加到末尾
  - 插入当前光标附近
  - 放弃
- 保存前自动保留版本

**这一项是 P0。**

这是当前写作体验里最影响可用性的点之一。

#### 4. 消灭剩余原生 `window.confirm`

当前编辑器主路径已经改到 `ConfirmDialog`，但模型配置页仍然残留浏览器原生确认。

本阶段应统一所有确认交互：

- 删除确认
- AI 覆盖确认
- 页面切换确认
- 危险操作确认

#### 5. 给编辑器补最基本的写作工具

本阶段只做高价值基础项：

- 章节目标字数
- 当前字数 / 完成度
- 最近保存时间
- 更明确的草稿 / 已完成状态
- 更清晰的“上次 AI 生成”反馈

先不引入复杂富文本。

#### 6. 修正文档口径

同步更新至少以下文档：

- `README.md`
- `docs/STATUS.md`

把已经过时的内容修正掉，例如：

- `lint` 状态
- `verify` 状态
- 测试数量
- CI 当前覆盖范围

### 2 周交付定义

到第 2 周结束时，应达到以下状态：

- 用户能从“作品列表 -> 小说详情 -> 角色/世界观/大纲 -> 编辑器”顺畅进入写作。
- AI 起草不再直接覆盖正文。
- 编辑器中的关键危险操作不再使用原生确认框。
- 文档口径与当前代码事实基本一致。

### 2 周验收标准

- 至少新增 1 个小说详情页和 3 个项目信息页。
- 编辑器支持候选稿处理，不再默认直接落正文。
- 相关 UI 有最小测试覆盖。
- 所有现有 `verify` 继续通过。

## 四、4 周目标：把“编辑器 MVP”升级成“创作工作台 v1”

### 阶段目标

在 2 周成果上，补齐真正的工作台结构，让产品开始接近 `design.md` 中设想的形态。

### 本阶段必须完成的事

#### 1. 上线真正的 Dashboard / 工作台首页

新增首页工作台，展示最有价值的信息，而不是做一个空 dashboard。

建议只放高频信息：

- 最近编辑的小说
- 待继续章节
- 最近一次 AI 生成记录
- 失败的后台任务
- 当日 / 当月用量概览
- “下一步建议”

#### 2. 建立章节管理页

从单纯的左侧章节树，升级成章节管理视图：

- 按卷分组
- 支持草稿 / 完成筛选
- 显示字数、更新时间、状态
- 可快速进入编辑
- 能看到“需要刷新摘要 / 索引”的状态

#### 3. 把 Beat Sheet / Outline Agent 接进主写作流程

当前 Beat Sheet API 已存在，但还不是主交互。

4 周时应形成标准闭环：

1. 先生成本章节拍
2. 用户确认或编辑节拍
3. 基于节拍生成正文
4. 结果进入候选稿，而非直接覆盖

这样大纲 Agent 才算真正进入产品，而不是只停留在 API 层。

#### 4. 补“生成历史”视图

需要一个可见的历史层，而不是只看章节版本：

- 最近 AI 生成记录
- 生成类型：起草 / Critic / State Diff / 大纲
- 时间、章节、状态、耗时
- 失败记录可见

这会显著提高用户对 AI 行为的可理解性。

#### 5. 补写作可靠性可见性

围绕 jobs、summary、retrieval、state diff 增加用户可见反馈：

- 当前章节是否已完成摘要
- 是否已完成索引
- 是否有失败 job
- 检索是否成功
- Story State 是否待确认

当前底层能力已经在，但用户面仍然较黑箱。

#### 6. 把 E2E 纳入 CI

这是 4 周内必须完成的工程动作。

至少将以下两条链路纳入 CI：

- onboarding 主链路
- editor failure 保护链路

如有时间，再补：

- 登录 / 注册最小链路
- 候选稿写作链路

### 4 周交付定义

到第 4 周结束时，产品应该从“单页编辑器产品”升级为“有项目管理层、有章节管理层、有 AI 工作流层”的创作工作台 v1。

### 4 周验收标准

- 存在 Dashboard 页面。
- 存在章节管理页与小说详情页。
- Beat Sheet 进入主写作流程。
- 存在生成历史页或等价视图。
- E2E 至少两条链路纳入 CI。

## 五、8 周目标：把长篇能力和产品可靠性推到可持续使用水平

### 阶段目标

把现有的 Agent / RAG / 摘要 / 状态系统，从“能工作”推进到“可信、稳定、可长期写长篇”的水平。

### 本阶段必须完成的事

#### 1. 做长篇记忆可靠性增强

重点不是继续新增概念，而是把已有链路做稳：

- 改稿后 summary/index/state 刷新链路更明确
- dirty 状态更细粒度
- 检索失败更显式
- 失败 job 支持可重试
- 检索结果来源展示更友好

#### 2. 补版本恢复与差异比较

当前只有版本列表，没有恢复。

8 周时至少要支持：

- 恢复某个历史版本
- 比较当前稿与历史稿差异
- AI 候选稿与正文差异对比

这样版本系统才真正能服务写作。

#### 3. 完成导出 / 发布中心 v1

从“一个菜单动作”升级为完整页面：

- Markdown / TXT 下载
- 后续扩展 docx / epub
- 导出预览
- 导出前审核状态说明
- 未来平台发布的预留位置

#### 4. 完成灵感库 / 素材库最小版

这一项不是当前最紧急，但 8 周时可以做最小落地：

- 灵感片段
- 角色片段
- 场景片段
- 可插入到当前项目

这会让平台从“生成器”更像“创作工作站”。

#### 5. 做工作台视觉统一和降噪

这部分要以 `design.md` 为目标，但要基于已存在页面逐步统一，而不是大翻修：

- 统一页面标题体系
- 统一卡片、空状态、错误态、生成中状态
- 降低“后台感 / 协议感”
- 强化“写作工具感 / 创作氛围”
- 编辑器优先、AI 次优先

#### 6. 为后续多人协作和分支创作打基础

8 周阶段不一定真正做多人协作，但应该把几个关键前置能力补到位：

- 版本恢复
- 候选稿合并
- 清晰的项目结构页
- 稳定的 ownership / 资源边界

否则之后多人协作会很难接。

### 8 周交付定义

到第 8 周结束时，产品应具备：

- 清晰的项目层、章节层、编辑层结构
- 安全可控的 AI 起草与候选稿机制
- 可见、可诊断的长篇记忆链路
- 可恢复、可比较的版本体系
- 更接近 `design.md` 的创作工作台体验

### 8 周验收标准

- 版本恢复可用。✅（M3.2）
- 候选稿 + diff 流程可用。✅（M3.2 — DiffView 已接 VersionsModal；候选稿与正文 diff 仍是 backlog）
- 记忆刷新 / 检索状态对用户可见。✅（M3.4 — retrieval SSE 携带命中片段，候选稿面板渲染 source/score/reason）
- 导出中心可独立使用。🟡（M3.3 — docx/epub 已接，仍以 ExportMenu 形式承载，独立页面 `app/(app)/novels/[id]/export/page.tsx` 在 backlog）
- 产品信息架构已不再依赖“编辑器侧栏承载一切”。✅（M1.1/1.2/2.1/2.3/2.4 累计交付）

## 六、明确不应在这 8 周优先做的事

以下事项有价值，但不应抢占前 8 周主线：

- 多人实时协作
- 分支创作系统
- 平台直发
- 计费支付
- Prompt Cache / 多模型复杂 Router
- 角色关系图可视化

原因不是这些不重要，而是它们依赖于前面的产品主干和版本机制先稳定。

## 七、推荐执行顺序

### 第 1 阶段（第 1-2 周）

- 小说详情页
- 角色 / 世界观 / 大纲独立页
- AI 候选稿机制
- 替换所有原生确认
- 编辑器基础写作工具
- 修正文档口径

### 第 2 阶段（第 3-4 周）

- Dashboard
- 章节管理页
- Beat Sheet 主流程接入
- 生成历史页
- job / retrieval / state 的可见性增强
- E2E 进入 CI

### 第 3 阶段（第 5-8 周）

- 长篇记忆可靠性增强
- 版本恢复 / diff
- 导出 / 发布中心
- 灵感库最小版
- 全站 UI 降噪统一

---

## 八、一句话判断

这不是一个“还要先补底层能力”的项目，而是一个“底层已经够用，必须开始认真做产品结构、写作工作流和长期可靠性”的项目。

接下来 8 周最重要的不是再多接几个 AI 能力，而是把已经存在的能力，组织成一个作者愿意每天使用的工作台。
